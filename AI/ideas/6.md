## 实验设计

### 模型

采用 **Qwen/Qwen3.5-0.8B** 作为底座模型。该模型是 24 层 hybrid 架构，适合做小规模 reasoning finetuning 与方法验证。初版仍在 `LinearOPSD` 路径中实现，不额外新开训练框架，也不新增新的 `conditioning_mode`；而是直接改现有 `linear_opsd` 这条训练路径。

### 训练数据

主训练集仍采用 **OpenThoughts-114k-math**。每条样本抽取：

\[
x = \text{problem}, \qquad y = \text{solution}
\]

其中 \( y \) 是完整 gold CoT。初版仍聚焦数学推理，不额外拆 answer-only 监督，也不引入额外 CE anchor，主目标仍是 rollout 段上的分布蒸馏。

备选小规模 sanity-check 数据集仍可保留：

- **GSM8K**
- **Bespoke-Stratos-17k**

但不作为主实验。

### 核心思路

这一版方法的目标，不再是让 teacher 只拿到被污染位置的 **local clean patch**；而是让 teacher 拿到**完整 gold solution**，再显式观察 student 在多个局部点出错后的 prefix 与后续 rollout，并据此给 student 提供“纠错式”的 privileged supervision。

整个训练流程可以直接概括为 3 步：

1. 先在 gold solution 上做一次前向，找出若干个高熵、非 style 的位置，并把它们替换成错误 token，构造 student corrupted prefix。
2. student 从这个 corrupted prefix 出发，沿自己的轨迹继续 rollout。
3. teacher 不单独 rollout，而是在看到 `problem + gt solution + 带 <corrupt> 标记的 student trace + 同一段 student rollout` 的前提下，对 student rollout 的每个位置提供 next-token supervision。

因此，训练中的两条条件上下文分别是：

- **student**：`problem + corrupted partial solution prefix`
- **teacher**：`problem + gt solution + 带内嵌 <corrupt> 标记的 student reasoning trace + student rollout`

其中 teacher 的 prompt 要明确说明：

- 已提供正确参考解答；
- 后面会提供 student 当前的推理轨迹；
- teacher 输入中会显式标出哪些位置是人工注入的 corruption；
- teacher 需要基于完整 gold reasoning 去判断 student 的轨迹偏差，并给出更合理的 next-token 分布。

teacher **不单独 rollout**。  
训练仍然沿用 OPSD / current `LinearOPSD` 的 shared-rollout 形式：student 先在错误 prefix 上实际 rollout，teacher 再在看到同一段 student rollout 的前提下做 teacher forward，并在这些 rollout token 上对 student 做蒸馏。

### teacher / student

- **student**：当前训练中的模型，参数记为 \( \theta_S \)
- **teacher**：teacher 策略可配置，参数记为 \( \theta_T \)

teacher 仍保留三种可切换策略：

- **dynamic teacher**：teacher 与当前 student 共享同一套实时参数
- **fixed teacher**：teacher 固定为初始策略
- **EMA teacher**：teacher 为 student 的指数滑动平均

默认仍优先跑 **dynamic teacher**，同时保留 `fixed teacher` 与 `EMA teacher` 作为实验开关。

在 rollout 位置 \( t \)，student 和 teacher 的条件分布写为：

\[
p_t = p_{\theta_S}(\cdot \mid x, \tilde y^S_{<t})
\]

\[
q_t = p_{\theta_T}(\cdot \mid x, y, \bar y_{<\tau}, \hat y_{\tau:t-1})
\]

其中：

- \( \tilde y^S_{<\tau} \) 是带多个错误点的 student prefix
- \( \bar y_{<\tau} \) 是将 `<corrupt>` 直接嵌入 student trace 后，teacher 实际看到的 prefix
- \( \hat y_{\tau:t-1} \) 是 student 在该错误状态下真实 rollout 出来的历史
- teacher 额外拿到完整 gold solution \( y \)

这意味着 teacher 的 privileged information 不再只是“被污染位置的 clean token”，而是“完整正确解答 + student 当前错误轨迹”。

### prompt 设计

这一版 `linear_opsd` 中，teacher prompt 的自然语言构造方式尽量参考原始 `opsd` 路径，而不是纯 token patch 形式。  
特别是 teacher prompt 的前半段，尽量保留原 `opsd` 里已经存在的重叠结构：

- `Problem: ...`
- `Here is a reference solution to this problem:`
- `=== Reference Solution Begin ===`
- `=== Reference Solution End ===`
- `After reading the reference solution above ...`

teacher 输入建议组织为：

1. `problem`
2. `gt solution`
3. `带内嵌 <corrupt> 标记的 student reasoning trace`
4. `student rollout`

保留字符串不要求真的加入 tokenizer 词表。初版只使用 `<corrupt>` 这一类显式标记，例如：

```text
<corrupt>
```

`<corrupt>` 不单独写成额外说明块，而是**直接嵌在 student trace 文本内部**，放在被替换错误 token 的前面。初版不引入真正 special token embedding，避免改 tokenizer / embedding matrix。

teacher 的 instruction 需要显式写明：

- 参考解答是正确的；
- student prefix 中某些位置是显式注入的错误点；
- 每个错误点都会用 `<corrupt>` 在 trace 内直接标出；
- teacher 应基于参考解答理解 student 的错误，并对 student 当前 rollout 位置给出更合理的 token 分布。

核心要求是：teacher 的输出空间仍然是“同一个 continuation 任务”的 next-token 分布，而不是改成另一个从头回答问题的任务。

更具体地说，teacher 不是重新“从零解题”，而是在更强上下文下，继续对 student 当前这条 reasoning trajectory 的后续 token 进行判断。

### corruption 与 rollout 方案

这一版不再使用 `5.md` 中“从同一样本其他位置随机抽 token span 替换”的 corruption 方案。  
新的目标是构造**更像模型真实 reasoning slip 的多个局部错误点**。

只污染 `solution`，不污染 `problem`。

#### 1. 基于 gold trajectory 做一次前向

对每条样本，先让模型在 `problem + clean solution` 上做一次 teacher-forcing 前向，得到 gold trajectory 上每个位置的 logits。

记第 \( i \) 个 gold token 为 \( y_i \)，对应预测分布为：

\[
\pi_i = p_{\theta}(\cdot \mid x, y_{<i})
\]

并计算该位置的熵：

\[
H_i = - \sum_v \pi_i(v)\log \pi_i(v)
\]

#### 2. 过滤 style tokens

不直接在所有高熵 token 上做 corruption，而是先过滤掉明显的 style / formatting token，例如：

- 纯空白或换行模式 token
- 标点 token
- 纯模板口头词，例如 “Let”, “So”, “Therefore” 一类不承载核心数学语义的 token
- 明显的 chat-template 结构 token

过滤后的目标是让被替换位置尽量位于**内容性 reasoning token** 上，而不是风格 token 上。

初版这里不追求完全一般化，允许用简单、显式、可检查的启发式规则。

#### 3. 选择多个高熵内容 token 作为错误位置

在过滤后的候选位置中，选取若干个高熵 token 位置：

\[
B \ge 1
\]

记这些位置按从左到右排序后为：

\[
s_1 < s_2 < \cdots < s_B
\]

初版允许 \( B \) 作为可配置超参数。  
这些位置不要求构成连续 span，而是多个离散 corruption points。

如果经过 style-token 过滤后，某条样本找不到足够多的候选位置，则**不 fail-fast**。  
初版采用更宽松的行为：

- 打印 warning；
- 使用当前能找到的候选位置数；
- 若一个合法位置都找不到，则该样本该步不注入 corruption，直接退化为 clean prefix 样本。

#### 4. 在每个位置替换为错误 token

对每个位置 \( s_i \)，用模型在该位置预测的一个**非 gt 的高概率 token** 来替换 gold token：

- 默认优先使用 top-1 wrong token；
- 若 top-1 恰好等于 gt，则取下一个最高概率且不等于 gt 的 token；
- 如有需要，也可保留 top-k wrong token 中采样的实验开关。

若某个候选位置在过滤后找不到合适的 non-gt replacement token，也不直接中断训练；而是：

- 打印 warning；
- 跳过该位置；
- 若最终所有候选位置都被跳过，则退化为 clean prefix 样本。

于是得到带多个错误点的 prefix：

\[
\tilde y_{<\tau}
\]

对应的 teacher-visible corrupted trace 记为：

\[
\bar y_{<\tau}
\]

其中 \( \bar y_{<\tau} \) 与 \( \tilde y_{<\tau} \) 表示同一条 student corrupted prefix，只是前者在每个被替换位置前面额外插入字符串 `\text{<corrupt>}`，用于给 teacher 显式提示这些位置是人工注入的错误点。

其中 \( \tau \) 是开始 rollout 的位置。由于 rollout 应该从**最后一次 corruption**之后开始，初版中令：

\[
\tau = s_B + 1 + o
\]

这里 \( o \) 是可配置的 rollout start offset，表示在最后一个错误 token 后保留少量 clean token，再开始 rollout。

#### 5. student rollout

student 从 `problem + corrupted prefix` 出发做实际 rollout，长度为 \( K \)：

\[
\hat y_{\tau:\tau+K-1}
\]

rollout 解码策略仍保留两种：

- **sampling rollout**
- **greedy rollout**

默认仍先采用 **sampling rollout**，greedy 作为对照。

### 训练时 teacher 的条件构造

在 rollout 的任意位置 \( t \in [\tau, \tau+K-1] \)，student 实际看到：

\[
\tilde y^S_{<t} = [\tilde y_{<\tau}, \hat y_{\tau:t-1}]
\]

teacher 则看到：

\[
\tilde y^T_{<t} =
\bigl[
x,\;
y,\;
\bar y_{<\tau},\;
\hat y_{\tau:t-1}
\bigr]
\]

其中：

- \( y \) 是完整 gold solution
- \( \tilde y_{<\tau} \) 是带多个错误点的 student prefix
- \( \bar y_{<\tau} \) 是将 `<corrupt>` 直接嵌入到 student trace 后得到的 teacher 可见版本
- student rollout 历史直接拼接在 teacher prompt 末尾

因此 teacher 的 conditioning 是“完整正确解答 + student 多点错误状态 + student rollout 历史”，而不是“对同一 prefix 做 token patch”。

这里最重要的约束是：

- student rollout 只有一条，就是 student 自己生成出来的那条轨迹；
- teacher 不生成另一条 teacher rollout；
- teacher 只是在更强的条件上下文下，对这同一条 student rollout 提供蒸馏信号。

### 输入示例

下面给出一个简化示例。设原题与 gold solution 为：

```text
Problem: Solve 2x + 3 = 11.

Gold solution:
2x + 3 = 11, so 2x = 8, hence x = 4.
```

假设选中了两个高熵位置，并将：

- `8` 错替成 `6`
- `4` 错替成 `3`

则 student 的 corrupted prefix 可以是：

```text
Problem: Solve 2x + 3 = 11.

2x + 3 = 11, so 2x = 6, hence x = 3.
```

若 rollout 从最后一个 corruption 点之后开始，并且 student 后续继续生成：

```text
 Therefore the answer is \boxed{3}.
```

则 student 实际输入示例为：

```text
Problem: Solve 2x + 3 = 11.

Please reason step by step, and put your final answer within \boxed{}.

2x + 3 = 11, so 2x = 6, hence x = 3.
```

teacher 输入示例为：

```text
Problem: Solve 2x + 3 = 11.

Here is a reference solution to this problem:
=== Reference Solution Begin ===
2x + 3 = 11, so 2x = 8, hence x = 4.
=== Reference Solution End ===

After reading the reference solution above, make sure you truly understand the reasoning behind each step. Do not copy it verbatim. Use it to identify and correct the corrupted parts in the student's reasoning trace below while continuing the same solution trajectory.

Here is the student's current reasoning trace:
2x + 3 = 11, so 2x = <corrupt> 6, hence x = <corrupt> 3.

 Therefore the answer is \boxed{3}.
```

这个 teacher 示例有意与原 `opsd` 路径保持较大重叠：

- 前半段仍然是 `Problem + Reference Solution`
- 中间仍然保留 `After reading ...` 这一类过渡指令
- 只是后半段不再要求 teacher 独立重新推理，而是给出 student 当前 reasoning trace，再把 student rollout 直接接在后面

这里 teacher 没有自己的独立 rollout；它只是基于更强的上下文，对同一段 student rollout 提供 next-token supervision。

### 损失函数

不使用 CE，只保留蒸馏损失。  
在 student 的实际 rollout 段上定义：

\[
L_{\mathrm{KD}}=
\frac{1}{K}
\sum_{t=\tau}^{\tau+K-1}
\Bigl[
\alpha \, \mathrm{KL}(q_t \,\|\, p_t)
+
(1-\alpha)\, \mathrm{KL}(p_t \,\|\, q_t)
\Bigr]
\]

其中：

\[
\alpha \in [0,1]
\]

是 forward / reverse KL 的混合系数。

总损失即：

\[
L = L_{\mathrm{KD}}
\]

初版主消融轴仍保留：

\[
\alpha \in \{1.0,\ 0.75,\ 0.5,\ 0.25,\ 0.0\}
\]

其中：

- \( \alpha = 1 \)：纯 forward KL
- \( \alpha = 0 \)：纯 reverse KL
- \( \alpha = 0.5 \)：对称混合

初版默认仍先用 **pure forward KL** 作为最重要对照。

### 实现约束

这一版最大的实现变化是：corruption 不再能完全在 data collator 内完成，因为它依赖一次真实模型前向来计算熵与 wrong-token 候选，而且还需要为多个 corruption point 生成显式标记信息。

因此在 `LinearOPSD` 的实现上，应遵循：

- 不新增新的 `conditioning_mode`
- 直接改现有 `linear_opsd`
- teacher prompt 组织方式尽量复用 / 参考现有 `opsd` 这条 prompt 风格
- entropy-based corruption 的生成逻辑应放在更靠近 trainer / rollout 的位置，而不是继续写成纯静态 collator 逻辑
- 当找不到合适的 corruption 位置或 replacement token 时，不中断训练，而是打印 warning 并退化处理

### 训练细节

- teacher 策略可在 dynamic / fixed / EMA 之间切换
- 每条样本每步可注入多个错误点
- 错误点从高熵、非 style token 中选出
- 错误 token 来自当前模型在对应位置的高概率非 gt 候选
- student rollout 从最后一个错误点之后开始
- teacher 输入中对每个错误点都做显式标记
- student rollout 解码方式可选 sampling 或 greedy
- teacher 与 student 共享同一条 student rollout 轨迹
- 蒸馏只在 rollout 的 \( K \) 个位置上计算
- `LinearOPSD` 路径仍以 full-vocabulary logit distillation 为主

### 评测

#### 1. 主评测：error-correction robustness

在 held-out math 样本上固定相同的多点错误注入协议，报告：

- clean prefix 下的解题准确率
- corrupted prefix 下的解题准确率
- recovery gap

\[
\text{recovery gap} = \text{clean acc} - \text{corrupted acc}
\]

- rollout 段平均

\[
\mathrm{KL}(q_t \,\|\, p_t), \qquad \mathrm{KL}(p_t \,\|\, q_t)
\]

以及混合损失

\[
L_{\mathrm{KD}}
\]

这一组指标仍是主指标，用来衡量模型是否更会从局部错误 CoT 中恢复。

#### 2. benchmark

- **GSM8K**
- **MATH-500**
- **MMLU-Pro**
- **IFEval**

其中前两者看数学泛化，后两者看 general capability / instruction-following 是否被明显破坏。

### 最小实验矩阵

主实验先只跑以下几组：

- **A**：\( \alpha = 1.0 \)，`rollout_decoding=sample`
- **B**：\( \alpha = 0.75 \)，`rollout_decoding=sample`
- **C**：\( \alpha = 0.5 \)，`rollout_decoding=sample`
- **D**：\( \alpha = 1.0 \)，`rollout_decoding=greedy`

然后再做以下小规模消融：

- wrong-token 选择：`top1-wrong` vs `topk-sample-wrong`
- style-token 过滤强度：宽松 vs 严格
- corruption 点数量：\( B \in \{1, 2, 4\} \)
- rollout 长度：\( K \in \{8, 16\} \)
- teacher 策略：dynamic vs EMA vs fixed

### 与 5.md 的主要区别

相较于 `5.md`，这一版的变化是：

- teacher 不再只拿 clean local patch，而是拿完整 gold solution
- teacher prompt 中显式加入 student 的错误 prefix 与 rollout
- 用保留字符串标记 student rollout 区段
- corruption 不再是随机 span 替换，而是基于熵的多点错误注入
- 方法目标从“local patch supervision”进一步转向“teacher-guided error correction”
